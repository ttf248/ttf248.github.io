昨年、SDKを設計しました。いくつかのイベントの処理とカプセル化を担当し、クラスインターフェースとして外部に提供します。サービス初期化時に、呼び出し元は対応するクラスを実装し、オブジェクトポインタをモジュールに渡します。 C11に触れたことがあり、好奇心から、これらのインターフェースをラムダ関数オブジェクトのコールバックで実装するとどうなるだろうかと考えた結果、純粋な仮想関数のインターフェース定義と比較して、より柔軟になるように思えた。 疑問が生じました。二つの異なる文法、パフォーマンスの観点から言えば、どちらがより高速でしょうか？コンパイラ原理は理解していませんが、コードを書いて試してみます。

まえがき

オンラインのウェブサイトで、さまざまなコンパイラやコンパイルパラメータを選択し、Linuxプラットフォーム上でコードを実行したり、対応するアセンブリコードを確認したりできます。

  • Wandbox.org：時々技術検証をする際に、ウェブページ上で短いコードの実行が手間を省くのに便利です。
  • Godbolt.org は、異なる色で異なるアセンブリコードに対応するコードを区別するため、ローカルのデバッガーよりも見やすくなります。

正文

標準委員会は文法の規則を策定し、コンパイルレベルでの実装方法は各々のコンパイラに依存しますが、ここで言及せざるを得ないのは、Microsoftのコンパイラが非常に優れていることだ。シンタックスシュガーは万能ではなく、コールバックインターフェースも多くはないので、lambdaを使用する方がより便利であり、空のコールバック関数インターフェースを定義する必要もない。コールバックインターフェースの種類が多くなる場合は、従来の仮想関数の方がビジネスインターフェースの統一に有利である。

  • Windowsプラットフォームですが、両者の性能はほぼ同等で、大きな違いはありません。
  • Linuxプラットフォーム、仮想関数とラムダの比較、単回で1.35ns増加

通常のビジネスシステム開発においては、このレベルのパフォーマンス損失は無視できる。lambdaを導入することで、設計においてより多くの利便性が得られる。特に多信号処理を行う場合、その効果が顕著になる。イベントトリガーがあり、ログを記録する必要がある場合は、ログオブジェクトの処理関数が必要となる。より多くのビジネス処理インターフェースが必要な場合は、vectorでlambdaオブジェクトを保存し、イベントトリガー時に順に呼び出す。これは、QTにおけるシグナルとスロットに似ており、ログ、監視、ビジネス1、ビジネス2などが完全に疎結合になる。

#include <iostream>
#include <chrono>
#include <memory>
#include <functional>
#include <atomic>
#include <string>

std::atomic_int64_t counter = 0;

// 定义回调接口
class UserInterface
{
public:
    virtual void name() = 0;
    virtual void full_name() = 0;
};

class User : public UserInterface
{
public:
    void name() {}
    void full_name() { counter++; }
};

void to_string(UserInterface* user)
{
    user->name();
    user->full_name();
}

using name_handler = std::function<void()>;
using full_name_handler = std::function<void()>;

class Test
{
    name_handler name_;
    full_name_handler full_name_;

public:
    void set_name_handler(name_handler name)
    {
        name_ = name;
    }

    void set_full_name_handler(full_name_handler full_name)
    {
        full_name_ = full_name;
    }

    void to_string()
    {
        name_();
        full_name_();
    }
};

int main()
{
    User user;

    auto start = std::chrono::high_resolution_clock::now();

    for (int i = 0; i < 1000000; i++)
    {
        to_string(&user);
    }

    auto end = std::chrono::high_resolution_clock::now();
    std::cout << "Counter: " << counter << std::endl;
    std::cout << "Time: " << std::chrono::duration_cast<std::chrono::microseconds>(end - start).count() << "us" << std::endl;

    counter = 0;
    auto name = []() {};
    auto full_name = []() { counter++; };

    Test test;
    test.set_name_handler(name);
    test.set_full_name_handler(full_name);

    start = std::chrono::high_resolution_clock::now();

    for (int i = 0; i < 1000000; i++)
    {
        test.to_string();
    }

    end = std::chrono::high_resolution_clock::now();
    std::cout << "Counter: " << counter << std::endl;
    std::cout << "Time: " << std::chrono::duration_cast<std::chrono::microseconds>(end - start).count() << "us" << std::endl;

    return 0;
}

資料を探しているとき、似たようなコードスニペットfunctionperformance.cppにたどり着きました。

#include <iostream>
#include <chrono>
#include <memory>
#include <functional>

using namespace std;
using namespace std::chrono;

class Base
{
public:
	Base(){}
	virtual ~Base(){}
	virtual int func(int i) = 0;
};

class Derived : public Base
{
public:
	Derived(int base = 10) : base{base}
	{

	}
	~Derived(){}

	virtual int func(int i)
	{
		return i*base;
	}
private:
	int base;
};

struct Func
{
	int base;
	int operator()(int i)
	{
		return i*base;
	}
	Func(int base) : base {base}
	{

	}
};
const int base = 10;
int calculate(int i)
{
	return base*i;
}

int main()
{
	const int num = 10000;
	Base *p = new Derived{10};
	int total = 0;
	auto start = high_resolution_clock::now();
	for (int i = 0; i < num; ++i)
	{
		total += p->func(i);
	}
	auto end = high_resolution_clock::now();
	std::cout<<"result: "<<total<<"\nvirtual call elapsed: \t"<<duration_cast<nanoseconds>(end-start).count()<<" nanoseconds.\n"<<std::endl;

	total = 0;
	start = high_resolution_clock::now();
	for (int i = 0; i < num; ++i)
	{
		total += calculate(i);
	}
	end = high_resolution_clock::now();
	std::cout<<"result: "<<total<<"\ndirect function call elapsed: \t"<<duration_cast<nanoseconds>(end-start).count()<<" nanoseconds.\n"<<std::endl;

	Func functor{10};
	total = 0;
	start = high_resolution_clock::now();
	for (int i = 0; i < num; ++i)
	{
		total += functor(i);
	}
	end = high_resolution_clock::now();
	std::cout<<"result: "<<total<<"\nfunctor call elapsed: \t"<<duration_cast<nanoseconds>(end-start).count()<<" nanoseconds.\n"<<std::endl;
	int base = 10;
	function<int(int)> lambda = [base](int i)
	{
		return i*base;
	};
	total = 0;
	start = high_resolution_clock::now();
	for (int i = 0; i < num; ++i)
	{
		total += lambda(i);
	}
	end = high_resolution_clock::now();
	std::cout<<"result: "<<total<<"\nlambda call elapsed: \t"<<duration_cast<nanoseconds>(end-start).count()<<" nanoseconds.\n"<<std::endl;
	return 0;
}

ここに2つのモード、普通の関数とラムダ関数が増えましたが、インターフェースコールバック方式と直接呼び出しの比較、パフォーマンス損失は桁違いの差異があります。ラムダ関数の性能は関数に近く、場合によってはラムダ関数の性能の方が優れています。コンパイラ原理に関しては知識の空白ですが、推測ではアクセスする変数のアドレスと関数が隣接しているため、CPU処理に有利であると考えられます。
