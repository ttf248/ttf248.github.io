금융 거래 시스템에 대한 테스트 투자액은 다른 시스템보다 훨씬 높지만, 복잡한 테스트 단계가 반복되고 ROI는 너무 낮습니다. 프로젝트와 인력의 교체로 인해 피할 수 없이 더 많은 통제 불가능한 요소들이 유입되고 있으며, 흔히 A 인터페이스 출력의 특정 필드를 수정하면 B 인터페이스 결과에 영향을 미치는 경우가 발생합니다. 매번 버전이 출시될 때마다 위험도 누적됩니다.

이론적 지식

  • 자동화의 가치를 어떻게 측정할 수 있나요?

자동화 테스트 사례 ROI = (수동 실행 시간) * (실행 횟수) / (개발 비용 + 유지 보수 비용)

  • 어떤 기능에 자동화 테스트가 필요합니까?

사용자가 자주 사용하는 기능, 자주 변경되지 않는 기능에 대해 자동화 테스트 코드를 작성하면 가장 큰 효과를 볼 수 있습니다

  • 자동화 테스트를 추진하기 위한 이 시점을 선택한 이유는 무엇입니까?

프로젝트 출시가 임박한 시점에는 적절하지 않습니다. 멀리 있는 물로 당장의 갈증을 해결할 수 없으며, 자동화는 장기적인 수익 모델입니다. 프로젝트는 이미 실제 운영 환경에 배포되었고 안정적인 릴리스 주기를 맞이하고 있으므로 지금이 가장 적합합니다.

프레임 선택

관련된 실무 경험이 부족한 상태에서 자동화 테스트라는 과제를 받았다면, 일반적인 시작은 다음과 같다: 검색 엔진을 열어 현재 시스템의 기술 스택에 사용할 수 있는 도구와 프레임워크를 찾고 사용 설명서를 훑어본다. 바로 적절한 도구를 찾으면 축하한다, 완벽한 시작이다.

먼저 잘못했다고 말씀드립니다. 관련 자료를 확인해 보니 없다는 게 아니라 프레임워크 자체가 너무 복잡하고, 배포 시 점유하는 리소스도 많습니다. 초보자 입문에는 작고 간결한 것이 필요합니다. 테스트 그룹 동료들과 상의 결과 기존 유닛 테스트 프레임워크를 활용하여 자동화 테스트 프레임워크로 캡슐화하는 Python 자체 구축 프레임워크가 언급되었습니다.

이 프로젝트의 디자인 아이디어를 참고하세요: https://github.com/wintests/pytestDemo

프레임워크가 왜 필요할까요?

서비스는 여러 개의 다양한 배포 환경을 가지고 있습니다. 개발 환경, 테스트 환경, 온라인 테스트 환경이 있으며, 프레임워크의 역할은 계층을 분리하고 테스트 사례와 데이터를 분리하는 것입니다. 각 환경에 따라 다른 사례 데이터를 구성하며, 물론 공통 데이터도 지원합니다.

핵심적인 논리는 자동화 활용률을 높이기 위한 것이다. 시나리오가 더 복잡해지고, 서로 다른 환경 간의 데이터는 일관성이 없고 아무런 관련이 없다면, 설정 사례 데이터를 추가할 때 label 태그를 늘려 현재 데이터가 지원하는 환경을 지정하면 된다.
