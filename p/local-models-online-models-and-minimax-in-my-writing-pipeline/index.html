本地模型、在线模型和 Minimax 混用以后，我对这条写作流水线的判断

2026年04月03日
阅读约 9 分钟
Codex (GPT-5)

写到这里，真正绕不开的问题其实只剩一个。

既然风格资产构建本身也不便宜，那我为什么后来会去接本地大模型，又为什么没有彻底抛弃在线版本，反而还把 Minimax 也接了进来？

我的答案很现实，因为这事从来都不是“谁更强”这么简单，而是成本、速度、稳定性、最终质量四件事一起算账。

系列导航：前文是 从手工复制到 blog-writer：我为什么一定要把博客写作流程做成 Skill 和 blog-style-suite：把风格数据、生成过程和 token 成本一起拆开。这是这个系列的第 3 篇。

本地模型最先解决的，是“敢不敢多跑几轮”

训练风格数据费 token，这个事说得再直白一点，就是你一旦开始多 provider 对比、多轮调试、多次重构 prompt，很快就会感觉在线模型的成本有点肉疼。

这时候本地模型的价值就出来了。

它不一定最强，但它有两个很实际的优势：

  • 你敢反复跑
  • 你敢拿来做预处理

尤其是风格构建这种低频但偏重的任务，本地模型天然更适合做“先来一遍粗加工”。哪怕效果不是最好，只要能把大部分重复劳动先吃掉，它就已经有价值了。

这个判断其实和正文生成不完全一样。正文生成更看重最终质感，风格构建更看重“能不能持续试错”。

但本地模型的问题也很明显

我不太认同那种一上来就把本地模型吹成万能替代品的说法。

本地模型的真实问题，主要就三件事：

  • 遵循度不一定稳定
  • 长上下文和复杂压缩任务容易飘
  • 同样一份输入，输出上限通常还是不如强在线模型

这也是为什么后来 blog-style-suite 没有把 provider 写死，而是保留了多 provider 结构。

你从现在的目录就能看出来，suite 会分别生成：

  • lm-studio-gemma.runtime.json
  • lm-studio-qwen.runtime.json
  • minimax-m2.runtime.json

最后真正给 blog-writer 生效的，不是它们全部，而是人工挑一份发布到 .agents/data/blog-writing/published.runtime.json。

这个设计我觉得很对。因为它承认了一件事：provider 不是宗教选择，而是结果选择。

为什么又接了 Minimax

很多时候，在线模型真正的价值，不是“全流程都在线跑”，而是作为质量基准。

我后来接 Minimax，一方面当然是因为它本身有可用性，另一方面也是因为我需要一个在线参照系。只有把本地版本和在线版本放到同一套数据结构、同一套输出契约里，差异才是可比的。

这个差异主要看三件事：

  • author_style 压出来够不够稳
  • samples 选得准不准
  • 当前 runtime 真拿去写稿时，味道是不是对

如果没有统一 pipeline，这个对比其实很虚。你今天换个 prompt，明天换个输入长度，后天又换个输出格式，最后根本分不清到底是模型差异，还是实验条件变了。

而现在 blog-style-suite 把 scan、build、runtime 契约都固定以后，本地模型和在线模型的差异，才终于能落到同一张考卷上。

我对这条流水线的最终判断

写到最后，我反而觉得这整件事最有价值的，不是“接了多少模型”，而是慢慢形成了一种更现实的分工方式。

1. token 富裕时，生吃原文依然有效

这事没必要回避。只要预算足够，直接带更多历史文章、带更多上下文，让强模型现场理解，效果通常不会差。

所以我不会把预处理神化成唯一答案。

预处理的意义，不是证明“生吃不行”，而是证明在长期工作流里，不是每次都值得生吃。

2. 高频环节必须轻

真正天天要跑的，是写稿，是起草，是补事实，是落盘。

这些环节如果还背着巨大的历史上下文，成本和等待时间都会越来越难看。所以运行时读取压缩后的 runtime，是非常合理的。

3. 低频环节可以重，但要有替代方案

风格构建、provider 对比、数据重建，这些低频动作可以允许更重一点。但也正因为它们重，所以才更需要：

  • 本地模型做试错
  • checkpoint 做恢复
  • 缓存做复用
  • 多 provider 做横向比较

这四件事加起来，才让它成为一条能长期维护的流水线，而不是一次性的实验品。

从 blog-writer 到 blog-style-suite，这里面真正长出来的东西

如果只看名字变化，很容易以为这只是工具越来越多。

但我觉得真正长出来的，其实是下面这套方法：

  • 高频任务和低频任务分层
  • 运行时数据和构建时数据分层
  • 本地规则和模型判断分层
  • 私有 skill 和仓库级工具分层

这几层一旦拆开，整个系统就稳定很多。哪怕后面再接新的本地模型，或者再换新的在线 provider，也只是替换其中一层，而不是把整套写作流程推倒重来。

这也是为什么我越来越觉得，AI 工作流真正值得写的，不是“某个模型强不强”，而是你有没有把那些真正反复发生的劳动，拆成一套能算得清账、换得了部件、跑得动很久的系统。

从这个角度看，blog-writer 和 blog-style-suite 的演化，写到最后已经不只是博客写作的事了。

它更像一个很小的样本，说明一件事：当 AI 真正进入生产环境以后，最后比拼的往往不是一次回答的惊艳程度，而是谁先学会把上下文、成本、数据和流程放到同一张表里一起管理。

写作附记

原始提示词 

本次的内容比较多，拆分成系列文章：去年就有很多稿子是通过大模型写的，那会是自己写个大纲或者问题清单，然后AI出稿子，复制内容到本地 md 文档，填写头信息，标签信息、发布文章；近期 codex 用了很多，发现 codex 里面的联网搜索能力很强，那我是不是能写个 skill，将这些事情自动化，此时诞生了 skill blog-writer 的第一稿，我还想着让 AI 学习我以前文章的风格，这就导致 blog-writer 运行的时候，很费 token，后续我针对 blog-writer 进行了多个版本的优化，拆分了 数据模块，数据生成的模块，原本数据生成的模块还是独立的 skill，写着写着，我就发现，更适合做成 Python 项目，此时就有了 blog-style-suite，然后我又发现，训练风格数据，也是比较费 token，我就想着用本地的大模型，对接了本地的大模型，我又想到了对比下本地大模型和在线版本的区别，又对接了 minimax；blog-style-suite 和 blog-writer 的演化历史可以分析的 git 提价记录。顺带基于本地 blog-writer、blog-style-suite 的代码，可以讲讲里面的设计思路，是如何做到了节约 token，数据结构是如何设计的，核心的设计思路。Token 富裕完全能生吃历史文章，预处理能节约很多 token

写作思路摘要

  • 最后一篇不再重复前两篇的结构，直接把 provider 分工、本地模型位置和最终判断讲透。
  • 核心不是评测某个模型，而是解释为什么必须保留多 provider 和人工发布位。
  • 保留“token 富裕可以生吃，预处理不是唯一答案”这个判断，避免把工程方案写成宗教立场。
  • 结尾回收到更一般的 AI 工作流方法论，让这组文章不只停在博客工具本身。
