Codex 的 goal 改的不是命令，是 agent 的收工条件

2026年05月27日
阅读约 9 分钟
Codex (GPT-5)
v7.2.0

Codex 的 goal 改的不是命令，是 agent 的收工条件

如果只把 Codex 的 goal 看成一条新命令，就会低估它。命令本身不稀奇，真正变化是它把“做到什么算完”从一句模糊提示，变成了 agent 可以围绕推进的目标。

普通提示词往往只告诉模型要做什么，比如修 bug、写文章、跑测试。goal 更像给一个持续工作的代理定义完成条件：什么时候可以停，停之前要拿到什么证据，遇到分支时应该优先推进哪条线。

这解释了为什么它能持续工作很久。长任务不是因为模型一次输出更长，而是因为它可以在目标、执行、验证、修正之间循环。做不到就继续查，测试失败就继续改，证据不够就继续补，直到接近 goal 描述的完成状态。

先别问能跑多久

看 goal，第一反应很容易是问时长：它是不是能后台跑很久，是不是能一直修到好，是不是不用我盯着。

这些问题会把方向带偏。

更关键的问题应该是：它继续工作的依据是什么？

OpenAI 的 Codex 文档把 /goal 描述成给当前线程挂一个持久目标。CLI 文档里，/goal <objective> 用来设置目标，/goal 可以查看，pause / resume / clear 可以控制它；use case 页面又把它放在 long-running、clear success condition、validation loop 这一类场景里。

这说明 goal 不是“再努力一点”的按钮。它更像把一个任务验收单挂到了线程上。后面每一轮，Codex 都应该围着这张验收单判断：还差什么，能不能继续，什么时候该停。

一个太松的目标长这样：

/goal 帮我把前端迁到 Next.js

这句话的问题不是短，而是没有停机条件。Codex 可以迁路由，可以改数据获取，可以顺手重构组件，也可以一直补它认为应该补的东西。你很难判断它是还没做完，还是已经开始扩写任务。

一个更可用的目标应该长这样：

/goal 把订单后台从 React Router 迁到 Next.js App Router。登录页、订单列表、订单详情和下单页的视觉行为要和旧版一致；不要改 API 合约和数据库 schema；每完成一批页面都跑 npm run build、npm test 和 Playwright 关键路径。只有这些验证都通过，才算完成。

多出来的不是客套话，而是四样东西：

  • 目标：要迁哪些页面。
  • 边界：API 和数据库不能顺手动。
  • 验证：构建、测试、Playwright 都要跑。
  • 停机：这些都通过，才算完成。

goal 抬高的就是这四样东西。

真正的新抽象

普通 prompt 更像“下一轮请你做什么”。你说一句，agent 做一轮，跑不跑测试、要不要继续、失败算不算阻塞，很多时候还要靠下一轮对话补。

goal 更像“这次任务的完成口径是什么”。它把控制权从一句临时请求，挪到一个更稳定的目标上。

这会改变任务写法。

以前我们常常这样补充：

再跑一下测试。
这个样式不算过。
别动接口。
构建失败先修，不要继续加功能。

这些话不是杂音，它们本来就是任务的一部分。只是过去它们散落在多轮对话里，现在应该提前进入 goal。

所以我会把可用的 goal 拆成四行：

目标：最终要出现什么用户可见结果。
边界：哪些文件、接口、数据、视觉或行为不能乱动。
验证：用哪些命令、测试、截图、评测或人工检查作为证据。
停机：全部满足时停止；遇到哪些权限、事实、产品判断时暂停。

写到这里，goal 就不只是命令了。它开始像一个小型工程契约。

为什么它能持续工作

Codex 能在一个 goal 下持续推进，不是因为一次回答被拉长了。

OpenAI 的长时任务文章把关键链路说得很清楚：长期工作依赖一个循环。大致是计划、改文件、跑工具、观察结果、修失败、更新状态，然后继续下一轮。

这个循环里，最重要的不是“模型更有耐心”，而是真实反馈一直在把任务拉回正轨。

构建失败会告诉它哪段迁移没过。

测试失败会告诉它哪个行为被破坏。

截图差异会告诉它页面不是“看起来差不多”。

eval 失败样本会告诉它 prompt 优化没有打到问题上。

如果 goal 里写清了验证方式，Codex 每轮就不是在凭感觉说“应该好了”，而是在追证据。证据没回来，就继续修；证据都回来，才有资格停。

这也是为什么 goal 适合迁移、重构、prompt eval、原型这类任务。它们不一定复杂到不可控，但都有一个共同点：一轮做不完，而且完成不能只靠主观判断。

反过来，有些任务不适合直接丢给 goal。

比如：

/goal 想一个更高级的产品方案

这句话没有验证，没有边界，也没有停机条件。它可能跑很久，但跑久不代表有用。先把它收束成“产出 3 个方案，覆盖哪些约束，用什么标准筛掉，什么时候停”，它才有资格变成 goal。

Claude Code 也在处理同一件事

Claude Code 也有 /goal，而且官方文档说得更直白：用户设置的是 completion condition，Claude 会跨 turn 持续工作，直到条件满足。文档还说明，每一轮结束后会检查条件是否成立；如果不成立，就继续下一轮。

这说明两家终端不是偶然撞了一个命令名。它们都在处理长任务里的同一个问题：

上一轮结束后，谁来判断还要不要继续？

/loop 这类能力偏时间驱动，hook 偏自定义规则驱动，goal 偏条件驱动。条件驱动的关键，不是“自动”，而是“有一个明确的完成条件在持续约束下一轮”。

OpenAI Codex 这边公开文档没有把内部 evaluator 细节完全摊开，所以不该替它脑补实现。但能确定的是，goal 的使用价值不在命令名，而在它迫使使用者提前写清楚完成口径。

该被改变的是写目标的人

goal 很容易被包装成“agent 可以自己干很久”。这句话听起来爽，但真正落到工作里，反而会害人。

越想让 agent 自己跑，越要把目标写窄。

越想少盯过程，越要把验证写实。

越想让它别跑偏，越要把边界写清楚。

所以 Codex 的 goal 不是把人从流程里拿掉。它只是把人的责任前移了。你不再每一轮提醒它“这个不算完成”，而是在一开始就把“什么算完成”写成它后续必须服从的条件。

这才是 goal 真正值得关注的地方。

不是多一条命令。

不是更长时间的自动工作。

而是终端 agent 开始把“做到什么算完”抬成一等公民。跑得久只是表象，知道凭什么停，才是能力。
