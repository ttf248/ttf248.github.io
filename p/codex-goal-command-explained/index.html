Codex 的 goal，不是多一条命令，是把“做到什么算完”抬成一等公民

2026年05月27日
阅读约 10 分钟
Codex (GPT-5)
v6.3.0

最容易把 Codex 的 goal 讲浅的方式，是从 slash command 开始讲。

一旦从命令讲起，文章很快就会滑成清单：/goal <objective> 怎么写，怎么看状态，怎么暂停恢复，Claude Code 有没有同名能力，OpenAI 又给了哪些长任务案例。信息都对，但读完以后只剩一个印象：原来 Codex 可以多跑一会儿。

这不是重点。

goal 真正新的一层，是把以前藏在提示词里的“做到什么算完”，单独拎出来，变成 agent 工作时必须持续对齐的对象。它不是让 Codex 更有耐心，而是先问清楚一件事：这次任务到底凭什么停。

不是加时按钮

普通 prompt 解决的是“下一轮帮我做什么”。你说“把这个页面迁到 Next.js”，Codex 可以给计划，可以改文件，也可以跑测试。但这一轮结束以后，它是否该继续、继续到哪里、哪些失败必须先修，往往还要靠你在下一轮补。

goal 改的是这里。

它把任务从一句请求，改成一个挂在线程上的完成条件。官方文档里强调的不是“写一个更长的 prompt”，而是把 objective、success criteria、constraints、validation 这些东西提前写清楚。换成人话，就是四个问题：

  • 这次要交付什么结果？
  • 哪些边界不能顺手改掉？
  • 用什么证据证明已经推进？
  • 满足什么条件就停，什么情况必须暂停？

所以，一个含糊的 goal 长这样：

/goal 帮我把前端迁到 Next.js

一个能工作的 goal 应该更接近这样：

/goal 把订单后台从 React Router 迁到 Next.js App Router。登录页、订单列表、订单详情和下单页的视觉行为要和旧版一致；不要改 API 合约和数据库 schema；每完成一批页面都跑 npm run build、npm test 和 Playwright 关键路径。只有这些验证都通过，才算完成。

多出来的不是啰嗦，是停机条件。

没有这些条件，agent 很容易把“继续努力”误解成“继续改更多东西”。有了这些条件，它每一轮都能问自己：我离完成还差哪条证据？

控制权变了

以前我们把很多控制权塞在对话里。

“再跑一下测试。”

“这个不算完成，视觉还不对。”

“别动接口层，只修页面。”

“构建过了再说。”

这些话不是杂音，它们其实是任务契约的一部分。goal 的价值，是把这些契约从一轮轮补充，提前变成一块稳定的工作面。

这会改变 Codex 的行为方式。它后面每次计划、改代码、跑命令、看日志、修失败点，都不是在追一个抽象的“帮我做好”，而是在追一个更可验收的状态：

done = 目标结果出现 + 禁止边界没被破坏 + 验证证据通过

这里最关键的是第三项。goal 不应该只写愿望，最好写证据。

“页面正常”不如“Playwright 跑完 login -> orders -> detail -> checkout”。

“代码质量可以”不如“npm test 和 npm run build 都是 0 退出码”。

“性能别太差”不如“列表页首屏接口不新增串行请求，Lighthouse performance 不低于旧版 5 分以内”。

一旦证据写清楚，Codex 的长时间工作才有意义。否则它只是长时间漂在一个模糊目标附近。

为什么能跑久

goal 能跑久，不是因为它把一次回答拉得更长。

真正的运行方式更像一个循环：

  1. 先根据目标拆下一小步。
  2. 改文件或跑工具。
  3. 观察测试、构建、截图、日志或 diff。
  4. 修掉失败点。
  5. 更新当前状态。
  6. 再拿 goal 判断下一轮还要不要继续。

OpenAI 在长时任务材料里反复强调的，也是这种 plan -> implement -> validate -> repair 的循环。长任务不是靠模型把所有上下文硬背住，而是靠真实工具反馈、外置状态和持续验证把方向拉回来。

这也是为什么 goal 更适合迁移、重构、原型、prompt eval 这类任务。

它们有三个共同点：

  • 一轮做不完。
  • 做完不是靠主观感觉判断。
  • 中间失败可以被测试、构建、截图或评测样本暴露出来。

反过来，如果任务本身没有可验证的完成条件，比如“帮我想一个更有高级感的方案”“把这个产品方向再研究研究”，硬塞进 goal 未必更好。它可能会跑很久，但跑久不等于有效。

这类任务要先收束成可验收对象。比如不要写：

/goal 优化客服 prompt

而是写：

/goal 优化 prompts/support-triage.md。先看 eval 失败样本，只做最小 prompt 修改；每次修改后跑 pnpm promptfoo eval。达到 92% 通过率，或者继续提高会触及产品/政策判断时停止。

后者才像一个 goal。它不仅告诉 agent 要做什么，也告诉它什么时候该停、什么时候不该硬冲。

Claude Code 的同名能力

Claude Code 也有 /goal。这一点比“某家有没有类似功能”更有意思。

Claude Code 文档把它说得更直白：用户设置的是 completion condition，系统会在轮次结束后检查条件是否满足；如果没有满足，就继续下一轮。它还把 /goal 和 /loop、Stop hook 这些能力区分开：/goal 是条件驱动，/loop 更偏时间驱动，hook 则是自定义规则驱动。

OpenAI Codex 的公开文档没有把内部 evaluator 细节摊得那么开，所以这里不该强行补一套“它一定怎么判断”的实现故事。能确定的是，两家都在把同一个东西抬高：

不是让 agent 多说几句，而是让 agent 每一轮结束后都面对一个更清楚的问题：

完成条件满足了吗？

这就是为什么 /goal 这个名字会同时出现。终端 agent 走到长任务以后，真正缺的不是又一个自动化开关，而是一个能长期约束行为的完成口径。

怎么写才不跑偏

我现在会把一个好 goal 拆成四行，而不是一句话：

目标：完成什么用户可见结果。
边界：哪些文件、接口、数据、视觉或行为不能乱动。
验证：用哪些命令、测试、截图、评测或人工检查作为证据。
停机：全部满足时停止；遇到哪些不确定性、权限、产品判断时暂停。

这四行看着普通，但它们决定了 agent 后面是不是有资格自驱。

如果只写目标，Codex 可能会补太多。

如果没有边界，它可能顺手重构不该重构的地方。

如果没有验证，它只能靠语言判断“应该好了”。

如果没有停机，它会在“再优化一点”和“已经可以交付”之间摇摆。

所以 goal 不是把人从流程里拿掉。恰恰相反，它要求人在开跑前把验收标准写得更像工程契约。人少盯每一轮，但要更认真地定义什么叫完成。

留一个边界

goal 很容易被包装成“让 agent 自己干到天荒地老”。这说法听着刺激，但会误导使用方式。

真正可用的长任务，通常不是无限开放的。它应该有清楚的仓库边界、验证命令、阶段状态和失败退出口。越是想让 agent 跑久，越要把停机条件写窄。

所以这篇最后只留一个判断：Codex 的 goal 值得关注，不是因为它多了一条命令，也不是因为它能连续工作很长时间，而是因为它把“做到什么算完”从聊天里的临时补充，抬成了 agent 工作流里的第一等对象。

以后终端 agent 真正拉开差距的地方，也会越来越像这里：目标怎么写，证据怎么拿，失败怎么停，进度怎么审计。

跑得久只是表象。知道凭什么停，才是能力。
