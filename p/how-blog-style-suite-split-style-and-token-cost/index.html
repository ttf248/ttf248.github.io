AI 写博客这件事，后来还是得做成工程（二）：blog-style-suite 怎么把风格学习和 token 成本拆开

2026年04月03日
阅读约 12 分钟
Codex (GPT-5)

如果 token 足够，最省脑子的办法其实很粗暴：把历史文章直接塞给模型，让它自己学。

问题在于，这种办法只适合偶尔来一篇，不适合反复写。

这个系列想讲的，是博客写作这条 AI 流水线怎么一步步长出来。上一篇写的是 blog-writer 为什么会长出来 。这一篇往前走一步，重点不是“文风像不像”，而是风格学习这件事为什么最后必须拆成独立的数据工程。下一篇会接着写 本地模型、在线模型和 Minimax 最后怎么分工 。

一开始最自然的想法，就是直接喂历史文章

我当时最直接的想法，其实跟很多人一样。

既然想让 AI 学我的写法，那就把以前的文章喂进去。你读得越多，不就越像吗？

这套思路在 token 富裕的时候完全成立。你甚至都不用设计什么数据结构，文章原文就是最完整的风格资产。

问题是，写作不是一次性的。

一旦进入重复调用，这套方式马上就会暴露成本：

  • 每次都要重复传大量历史文章
  • 当前文章自己的大纲、事实、链接也要占上下文
  • 系列模式下还得额外带标题规划和内链关系
  • token 成本和延迟会一起往上走

更现实一点说，你喂得越多，模型未必越稳。因为真正跟当前任务有关的风格信号，只是历史文章里的很小一部分。剩下的大量正文，其实都在消耗上下文预算。

所以我后面越来越确信一件事。风格学习不能总在写稿当下做。写稿时应该消费压缩后的结果，而不是每次都从零重新咀嚼历史文章。

这也是为什么数据模块和数据生成模块必须拆开

blog-writer 后来一边写一边加，很快就发现，风格学习这件事如果继续塞在 skill 里，整体会越来越别扭。

原因不复杂。

skill 更适合做运行时约束，告诉写作者：

  • 去哪里拿风格数据
  • 该怎么组织标题和开头
  • 哪些信息能看，哪些信息不能看
  • 最终 Markdown 应该怎么落盘

但风格数据本身的生成，不是这个问题。

风格数据生成涉及另外一套东西：

  • 扫描历史文章
  • 提取 frontmatter、开头、结尾、小标题
  • 先做筛选，再做 lane 分类
  • 再从候选集中精选样本
  • 最后压缩成运行时能消费的结构

这已经更像一个数据构建项目，而不是一条 prompt 或一个 skill 了。

所以后面那条线就很自然地分岔了。运行时继续保留 blog-writer，数据构建这一侧慢慢拆成独立模块，最后收成了 blog-style-suite。

从 git 提交也能看得很清楚。84a06b5 这个节点，基本就是一个分水岭。它把原来的 blog-style-maintainer skill，收成了仓库级 CLI 工具。换句话说，风格维护从“会话内做一件事”，变成了“项目里维护一条管线”。

省 token，不靠玄学，靠的是预处理和批量化

blog-style-suite 后面的设计，我觉得最值钱的地方，不是模型选型，而是它终于把“哪些东西必须在运行时做，哪些东西应该提前做”分开了。

现在这套结构很清楚：

  • scanner.py 负责扫描 content/post
  • 先从文章里抽 frontmatter、开头段落、小标题、结尾这些锚点
  • 再用启发式规则做筛选和 lane 分类
  • builder.py 再做评分、精选、作者风格压缩、样本卡片渲染
  • 最后产出运行时只需要的 runtime.json

也就是说，真正 expensive 的事情，不再按“每篇文章一次调用”去做了，而是尽可能提前压缩。

bc4b950 这个提交把这条线讲得非常直白。AI 调用次数从每个 provider 大约两千次，收到了最多五次。

这个量级差异，不是小优化，是思路变了。

以前的问题写法，更像是“让模型参与每一篇历史文章的判断和压缩”。

后来的写法，是：

  • scan 阶段纯启发式，0 次 AI 调用
  • 每个 lane 最多 1 次批量精选和打标签
  • 全局作者风格再压缩 1 次

冷启动算下来，一个 provider 最多 5 次。有缓存的时候，甚至可以做到 0 次。

这才是预处理真正值钱的地方。

现在这套数据结构，本质上就是在压缩真正有用的信号

运行时的 published.runtime.json 现在只有四块核心数据：

  • author_style
  • lanes
  • samples
  • writer_guide

我觉得这个切法很对。

因为它不是在复制历史文章，而是在保留真正会影响写稿的那部分信号。

author_style 保留的是整个人的底色，比如是不是判断前置、是不是先抓触发点、是不是喜欢把问题拉回现实约束。

lanes 解决的是局部增强。技术、金融、随笔、工具体验，这几类文章的推进方式本来就不一样，不拆 lane，文章会写得很平。

samples 更关键。它不是把原文塞回来，而是把每篇代表文章压成几张卡：

  • 这篇文章怎么起手
  • 结构怎么推进
  • 语气有什么特点
  • 明确不能学什么

这一步特别重要。因为你真正需要的，不是把旧文章原封不动读一遍，而是把“为什么它像我写的”抽成可复用的提示。

writer_guide 则是最后一道边界，明确告诉 blog-writer 应该先读什么、最多读多少，不该再去碰哪些数据。

这套设计看起来简单，实际很顶用。它把运行时上下文从“大量原文”压成了“少量高价值结构化信号”。

不是所有事情都该让模型做

这套项目我后来越做越明显的一个判断是，别把模型当成万金油。

能在本地用规则先做掉的，就先做掉。

比如：

  • 路径扫描
  • frontmatter 提取
  • <!--more--> 检查
  • 正文长度过滤
  • 引用块占比检查
  • heading、tag、category 的 lane 提示

这些东西本来就不值得浪费模型调用。让模型来做，不仅贵，还不稳定。

真正应该留给模型的，是那些“规则很难写死，但人能一眼看出来”的判断。比如某个 lane 里十几篇候选，哪几篇更能代表当前的写法；再比如全局作者风格到底该压成哪几类表述。

这就是 blog-style-suite 后来最像工程的地方。不是一味追求“AI 参与更多”，而是明确区分“AI 应该做什么”和“AI 不应该做什么”。

预处理不是为了省一点 token，是为了让写稿这件事可持续

很多人聊 token，会把它理解成成本优化。

我现在更倾向于把它理解成可持续性问题。

如果写作链路长期依赖大上下文生吃历史文章，那它就天然更贵、更慢、更不稳定。你每加一篇旧文章，后面每次写稿都要一起买单。

而预处理的好处就在这里。历史文章的重活脏活，提前做一遍就够了。运行时只读压缩后的 runtime，既省 token，也更稳。

所以我现在看 blog-style-suite，它的价值不是“又多了一个项目”，而是终于把博客写作这件事从一次性 prompt，推进成了分层明确的数据管线。

blog-writer 负责消费。

blog-style-suite 负责生产。

两边拆开以后，后面再谈本地模型、在线模型、Minimax，事情才真正好讨论。

原始提示词 

$blog-writer 本次的内容比较多，拆分成系列文章：去年就有很多稿子是通过大模型写的，那会是自己写个大纲或者问题清单，然后AI出稿子，复制内容到本地 md 文档，填写头信息，标签信息、发布文章；近期 codex 用了很多，发现 codex 里面的联网搜索能力很强，那我是不是能写个 skill，将这些事情自动化，此时诞生了 skill blog-writer 的第一稿，我还想着让 AI 学习我以前文章的风格，这就导致 blog-writer 运行的时候，很费 token，后续我针对 blog-writer 进行了多个版本的优化，拆分了 数据模块，数据生成的模块，原本数据生成的模块还是独立的 skill，写着写着，我就发现，更适合做成 Python 项目，此时就有了 blog-style-suite，然后我又发现，训练风格数据，也是比较费 token，我就想着用本地的大模型，对接了本地的大模型，我又想到了对比下本地大模型和在线版本的区别，又对接了 minimax；blog-style-suite 和 blog-writer 的演化历史可以分析的 git 提价记录。顺带基于本地 blog-writer、blog-style-suite 的代码，可以讲讲里面的设计思路，是如何做到了节约 token，数据结构是如何设计的，核心的设计思路。Token 富裕完全能生吃历史文章，预处理能节约很多 token

写作思路摘要

  • 这一篇把重点从写稿动作切到数据工程，核心回答“为什么必须拆模块”。
  • 直接把“token 富裕可以生吃历史文章，但长期不划算”放在前面，避免铺太长背景。
  • 用 scanner.py、builder.py、compressor.py 的职责拆分来解释数据结构，不写空泛架构图。
  • 特别点出 bc4b950 的批量化重构，因为这是节省 token 最有说服力的节点。
  • 结尾把“消费侧”和“生产侧”分开，为第三篇的模型分工做铺垫。
