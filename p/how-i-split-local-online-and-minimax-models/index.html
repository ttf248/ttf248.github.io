AI 写博客这件事，后来还是得做成工程（三）：本地模型、在线模型和 Minimax 最后怎么分工

2026年04月03日
阅读约 11 分钟
Codex (GPT-5)

风格数据这件事，一旦开始认真做，token 很快就会变成现实问题。

不是你想不想省，而是你如果不分工，这套东西根本跑不久。

这个系列前面两篇，已经把起点和拆分过程交代过了。第一篇写的是 blog-writer 为什么会长出来 ，第二篇写的是 blog-style-suite 怎么把风格学习和 token 成本拆开 。这一篇收口，重点不是再讲一遍架构，而是讲最后一个更现实的问题：本地模型、在线模型和 Minimax，到底该怎么分工。

训练风格数据，不值得每一步都烧在线模型

事情走到 blog-style-suite 以后，我的感受很直接。

风格数据构建这件事，跟写单篇博客完全不是一个成本级别。

单篇写稿，调用次数有限，资料也有限。风格构建不是。它要扫描历史文章、做筛选、做 lane 分类、跑精选、跑风格压缩，还要反复调试提示词和输出结构。

这个阶段如果全部压到在线模型上，代价会很不好看。

因为你并不是只跑一次。你会反复调：

  • 规则是不是该收紧
  • lane 分得对不对
  • 样本是不是选偏了
  • 风格摘要是不是太虚
  • 输出格式能不能更稳定

这些动作每来一轮，都是新的 token 成本。

所以后面我会接本地模型，这不是情怀，也不是为了“证明本地也能跑”，而是这类任务本来就更适合先在本地折腾。

本地模型更适合脏活、重活和反复试错

我现在看，本地模型最适合三类事情。

第一类，是脏活。

比如要不要继续加一个字段、某个 prompt 结构要不要改、样本摘要需不需要再压一点，这些事情都需要反复跑。你让在线模型一次次陪你试，心态上也别扭，账单上也别扭。

第二类，是重活。

风格数据构建天然就比单篇写稿更像批处理。扫描一批文章、压缩一批样本、重建一版 runtime，这些动作用本地模型更合理。速度不一定最优，但至少你可以多跑几轮，敢调整。

第三类，是对比。

你只有把本地模型和在线模型都接进来，才能真正看出差异。不是抽象地说谁强谁弱，而是具体到这条管线里：

  • 谁的 JSON 更稳
  • 谁的风格摘要更虚
  • 谁的 lane 样本选择更像人
  • 谁在同样 prompt 下更容易跑偏

这也是为什么后面 blog-style-suite 不再绑定单一 provider，而是逐步支持多模型配置。

在线模型更适合收口，不适合包办一切

这件事我后面越来越确定。

在线模型当然更强，很多时候输出质量也更好。但这不代表它该包办整条链路。

如果一条链路里，有大量本地规则就能处理的事情，或者有大量需要反复试错的事情，把它们一股脑都塞给在线模型，本质上是在用更贵的办法做并不需要那么贵的事。

所以这套分工后来就越来越明确了：

  • 扫描、提取、筛选、分类，这些尽量本地规则先做
  • 真正难写死的压缩和选择，再交给模型
  • 需要大量实验和迭代的阶段，本地模型优先
  • 最后要拿出来对比、发布、作为正式风格资产的一版，再看在线模型结果

这套分法很土，但很有效。

它背后的逻辑也不复杂。在线模型的价值，不该浪费在“能不能提取 frontmatter”“能不能统计 heading”这种问题上，而应该留给那些规则写不死、但又确实影响最终质量的地方。

Minimax 的意义，不只是多接了一个 provider

我后来又对接 Minimax，最开始当然也是想对比下本地模型和在线版本的差别。

但真跑起来以后，我觉得它更重要的意义，不只是“又多了一个模型可选”，而是它把 provider 选择这件事彻底从写稿流程里剥离出来了。

这一点很关键。

因为 blog-writer 运行时并不关心你背后试过几个 provider，它只认一个固定发布位：.agents/data/blog-writing/published.runtime.json。

也就是说，维护侧可以有很多版本：

  • 本地模型压出来的一版
  • 在线模型压出来的一版
  • Minimax 压出来的一版

但消费侧不需要跟着变。它只消费你最后挑中的那一版 runtime。

这个边界其实比“多 provider 支持”本身更重要。因为一旦消费侧和 provider 绑死，每换一个模型，blog-writer 的行为就会被带着一起漂。

现在这种做法就清楚得多：

  • blog-style-suite 负责多 provider 构建
  • 人工挑一版结果发布
  • blog-writer 只吃发布位

这条边界在文档和代码里都很明显。46f37e3 讲清楚的是发布边界，9f15199、66fbfb4 这些节点，则逐步把多模型配置和当前 provider 收敛进了项目配置里。

真正的分工，不是按模型强弱分，是按任务类型分

我现在不太喜欢那种笼统判断，什么“本地模型不行”“在线模型才是正解”。

这类话都太空。

真正有用的分法，应该按任务类型来。

像博客写作这条链路里，我现在更认下面这种分法：

  • 写单篇博客正文，尤其带联网搜索、事实补充、系列规划的，在线 agent 更合适
  • 扫描历史文章、重建风格数据、反复调提示词结构的，本地模型更合适
  • 需要做 provider 横向对比、又想兼顾速度和效果的，Minimax 这种在线兼容 provider 很适合拿来做中间层

你看现在 config.json 里的 provider 配置，已经是 openai-compatible 这一套了。这种抽象其实挺实用。它不是在强调某一家，而是在强调上层管线不想被底层模型接口绑住。

这就像 blog-writer 和 blog-style-suite 的拆分一样。真正稳定的系统，最好都别把变化最大的那一层直接绑死在运行时。

到最后，最值钱的不是模型，而是边界清楚

我现在回头看这一整条演化线，最深的感受反而不是“用了多少模型”，而是边界终于慢慢清楚了。

blog-writer 负责写稿和落盘。

blog-style-suite 负责扫描、构建、压缩和发布准备。

本地模型负责便宜地试错、重建和折腾。

在线模型负责更强的压缩和更稳定的最终候选。

Minimax 这种 provider，则把“对比”和“兼容”这件事接进来了。

这套分工一旦站稳，后面的事情就会简单很多。你不用再纠结“到底该不该全都交给最强模型”，而是反过来问一句：这一步到底值不值得用最贵的方式做。

很多时候，答案其实很明确。

所以我的结论也很直接。真正让这条写作流水线站住的，不是某个模型突然更强了，而是你终于愿意把不同层的问题拆开，让不同成本、不同能力的东西，去做各自合适的事。

写作附记

原始提示词 

$blog-writer 本次的内容比较多，拆分成系列文章：去年就有很多稿子是通过大模型写的，那会是自己写个大纲或者问题清单，然后AI出稿子，复制内容到本地 md 文档，填写头信息，标签信息、发布文章；近期 codex 用了很多，发现 codex 里面的联网搜索能力很强，那我是不是能写个 skill，将这些事情自动化，此时诞生了 skill blog-writer 的第一稿，我还想着让 AI 学习我以前文章的风格，这就导致 blog-writer 运行的时候，很费 token，后续我针对 blog-writer 进行了多个版本的优化，拆分了 数据模块，数据生成的模块，原本数据生成的模块还是独立的 skill，写着写着，我就发现，更适合做成 Python 项目，此时就有了 blog-style-suite，然后我又发现，训练风格数据，也是比较费 token，我就想着用本地的大模型，对接了本地的大模型，我又想到了对比下本地大模型和在线版本的区别，又对接了 minimax；blog-style-suite 和 blog-writer 的演化历史可以分析的 git 提价记录。顺带基于本地 blog-writer、blog-style-suite 的代码，可以讲讲里面的设计思路，是如何做到了节约 token，数据结构是如何设计的，核心的设计思路。Token 富裕完全能生吃历史文章，预处理能节约很多 token

写作思路摘要

  • 第三篇不再重复讲架构，而是把“模型分工”这个现实问题单独收口。
  • 核心不是证明谁更强，而是说明为什么不同任务该由不同成本层来承担。
  • 明确点出发布位和 provider 解耦，是为了把 Minimax 的意义讲到工程层，而不只是工具层。
  • 把本地模型定义成“脏活、重活、反复试错”的承担者，这样更贴近真实使用感受。
  • 结尾回到“边界清楚比单点更强更重要”这个总判断，作为整组文章的收口。
