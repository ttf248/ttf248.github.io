AI 写博客这件事，后来还是得做成工程（一）：blog-writer 为什么会长出来

2026年04月03日
阅读约 10 分钟
Codex (GPT-5)

去年写了不少 AI 稿子，那会最土的流程就是，自己先整理个大纲或者问题清单，让大模型把正文吐出来，然后再把内容复制到本地 md 文档里，补 frontmatter、标签、分类、标题，最后再发布。

这套流程不是不能用，是很烦。真正费时间的地方，不是正文，而是正文外面那一圈重复劳动。

这组文章想讲的，不是“AI 能不能写博客”，而是博客写作这件事为什么最后会越做越像工程。第一篇先把起点说清楚：blog-writer 到底为什么会长出来。后面两篇会分别接着写 blog-style-suite 怎么把风格学习和 token 成本拆开 ，以及 本地模型、在线模型和 Minimax 最后怎么分工 。

真正烦人的，不是写稿，是那一串机械动作

那会每次写文章，流程都差不多：

  • 自己先想清楚要写什么
  • 把问题拆给大模型
  • 把正文复制到本地
  • 手工补标题、日期、标签、分类
  • 检查 <!--more-->
  • 再确认目录、slug、链接有没有问题

说白了，这里面很多动作根本不值得人脑反复参与。人真正该做的，是想清楚这篇文章到底想说什么，事实锚点够不够，判断站不站得住。结果现实是，很多时间都耗在体力活上。

我后来重度用 Codex，一个很直接的感受就是，它的联网搜索能力上来以后，很多以前还得我自己补的资料，现在也可以交给 agent 处理。问题就变了。

以前的问题是，AI 能不能帮我出稿。后来变成了，我能不能把整套写稿流程都交给 AI，而且它还得按我的方式做。

blog-writer 的第一稿，就是从这个地方长出来的。

blog-writer 第一层价值，不是文风，是把契约写死

很多人一提博客写作 skill，第一反应就是学文风。这只说对了一半。

文风当然重要，但第一层真正解决的问题，是流程稳定。

如果没有 skill，每次写文章你都得重新解释一遍：

  • 该去哪里读风格资产
  • 哪些资料可以看，哪些资料不能看
  • 什么时候必须联网确认事实
  • frontmatter 里哪些字段不能漏
  • 输出路径该怎么生成
  • 如果写成系列，系列内链接怎么串

这些事情单看都不大，但你每次都靠 prompt 临时说，模型就一定会漂。今天忘了作者，明天忘了 <!--more-->，后天又把路径猜错了。那就不是“偶尔不稳”，而是这套流程本身没有契约。

所以我回头看 blog-writer 的早期提交，重点其实不是“写得更像我”，而是先把重复说了无数次的话收进工具里。

从 git 历史也能看出来这条线。991536a 是最早把博客写作助手 skill 放进仓库的节点，后面 bc384d9、5871b8a、1d49002 这些提交，基本都在做同一件事：把原来散在脑子里的流程约束，慢慢挪成固定规则。

这一步看起来不花哨，但很关键。因为博客写作不是一次性任务，它是高频、重复、长期的。高频任务最怕的，不是单次写不好，而是每次都重新猜一遍边界。

系列模式，其实也是写作契约往前走了一步

这个问题在系列文章里更明显。

单篇文章好歹只用管一篇自己的 frontmatter 和路径。系列文章不是。系列一多出来，马上就会多一堆问题：

  • 标题要不要统一
  • 日期怎么排，才能保证顺序稳定
  • 系列内链接怎么自动生成
  • 目录怎么规划，才能不靠手工猜
  • 三篇之间怎么避免互相重复

所以 1a5604e 这个提交很有代表性。它不是简单多了个脚本，而是把“系列写作”从单篇文章的重复调用，单独拉成了一种写作模式。write_post_series.py 做的事情很直接：先规划整组文章的目标路径，再在落盘的时候统一替换 relref。这个设计不复杂，但非常实用。

因为系列文章最大的问题，从来不是 AI 会不会写三篇，而是你能不能让它写完以后，三篇都落在正确的位置，还能互相跳转。

但只靠一个 skill，后面还是会撞到 token 墙

问题也恰恰出在这里。

blog-writer 第一稿出来以后，我很自然地又往前想了一步。既然流程已经能自动化了，那能不能顺带让 AI 学我以前文章的风格？

这件事听起来很合理，真正一做，马上就会撞到 token 成本。

最原始的办法其实最简单：把历史文章直接喂进去。只要 token 足够，当然能生吃。

问题在于，真实使用不是偶尔写一篇，而是反复写。你一旦把历史文章、大纲、事实资料、外部链接、系列规划全部塞进一个上下文里，成本和延迟都会越来越难看。

而且更尴尬的一点是，喂得越多，未必越像。因为跟当前文章真正相关的信号，反而会被大段历史文本淹掉。

我后来回头看，这其实就是 blog-writer 演化的分水岭。

前一阶段，重点是把“写稿动作”自动化。

后一阶段，重点就变成了另外一个问题：哪些信息应该预处理，哪些信息没必要每次写稿都重新喂一遍。

这也是为什么后面会出现 blog-style-suite 怎么把风格学习和 token 成本拆开。写到这一步，事情已经不再是“一个 skill 多补几条规则”那么简单了，而是风格数据、数据生成、运行时消费，这几个阶段本来就该拆开。

先把流程做稳，后面才有资格谈风格和模型

现在回头看，blog-writer 真正解决的是两件事。

第一件事，是把过去那一串重复劳动自动化。

第二件事，是把博客写作这件事从“每次重新解释一遍”变成“按固定契约执行”。

这两步如果没做，后面讨论风格学习、本地模型、在线模型、token 节约，其实都站不住。因为底层流程本身就是散的，你喂再多模型也只是让它在一个不稳定的框架里反复猜。

所以我的判断一直没变。blog-writer 第一稿看起来像在解决写作问题，实际上先解决的是工程问题。

只有这一步站稳了，后面的风格压缩、runtime 数据、本地模型对比，才有意义。

写作附记

原始提示词 

$blog-writer 本次的内容比较多，拆分成系列文章：去年就有很多稿子是通过大模型写的，那会是自己写个大纲或者问题清单，然后AI出稿子，复制内容到本地 md 文档，填写头信息，标签信息、发布文章；近期 codex 用了很多，发现 codex 里面的联网搜索能力很强，那我是不是能写个 skill，将这些事情自动化，此时诞生了 skill blog-writer 的第一稿，我还想着让 AI 学习我以前文章的风格，这就导致 blog-writer 运行的时候，很费 token，后续我针对 blog-writer 进行了多个版本的优化，拆分了 数据模块，数据生成的模块，原本数据生成的模块还是独立的 skill，写着写着，我就发现，更适合做成 Python 项目，此时就有了 blog-style-suite，然后我又发现，训练风格数据，也是比较费 token，我就想着用本地的大模型，对接了本地的大模型，我又想到了对比下本地大模型和在线版本的区别，又对接了 minimax；blog-style-suite 和 blog-writer 的演化历史可以分析的 git 提价记录。顺带基于本地 blog-writer、blog-style-suite 的代码，可以讲讲里面的设计思路，是如何做到了节约 token，数据结构是如何设计的，核心的设计思路。Token 富裕完全能生吃历史文章，预处理能节约很多 token

写作思路摘要

  • 第一篇先不急着讲 token，而是先把 blog-writer 出现的现实动机说清楚。
  • 重点保留“复制正文不难，重复劳动很烦”这条主线，避免写成工具介绍。
  • 用早期 git 节点把“流程契约化”这件事压实，而不是只讲感受。
  • 系列模式和 write_post_series.py 放在这一篇讲，是为了说明写作工程化已经从单篇走向整组输出。
  • 结尾把主线自然引到第二篇的 token 和风格数据拆分。
