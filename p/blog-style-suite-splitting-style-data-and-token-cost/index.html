AI 写博客这件事，后来还是得做成工程（二）：风格数据和 token 成本为什么必须拆开

2026年04月03日
阅读约 11 分钟
Codex (GPT-5)

blog-writer 解决的是写稿动作自动化，blog-style-suite 解决的就是另一层更现实的问题：风格学习这件事，不能每次写稿都重新烧一遍 token。

写到这里，skill 这条路其实已经不够了。它更像一条需要反复跑、反复调、还能换 provider 的构建流水线，所以最后才会长成 Python 项目。

上一篇我讲的是 blog-writer 是怎么从复制粘贴里长出来的。这一篇往前推一步，专门讲为什么风格数据、构建过程、运行时读取必须拆开。下一篇再收口到 本地模型、在线模型和 Minimax 该怎么分工。

最朴素的办法当然有用，但代价也最直接

先把话说死一点。Token 足够富裕的时候，直接生吃历史文章，当然是能跑的。

你把旧文章批量塞给模型，让它现场归纳风格，再结合当前题目起草正文，这件事在能力上没什么问题。甚至在样本还不多的时候，它还挺省事。

但它有两个很现实的副作用：

  • 每次写稿都要重复付费
  • 每次都要重复等待

而且样本越多，这个问题越重。你博客写得越久，历史文章越多，重复喂上下文这件事就越不划算。

所以我后来越来越明确一件事：风格学习和正文生成，应该是两个阶段。

风格学习是低频动作，正文生成是高频动作。低频动作可以重一点，高频动作必须轻。

这句话听上去像废话，但很多 AI 工作流一开始失败，恰恰就是因为把这两件事混在一起了。

为什么最后更适合做成 Python 项目

最开始数据生成模块其实还是往 skill 的方向写的，但写着写着就发现不对。

因为它处理的已经不是“单次任务说明”，而是一条完整的数据管线：

  • 扫描 content/post
  • 提取 frontmatter、正文开头、结尾、小标题、摘录
  • 判断哪些文章该入选，哪些该排除
  • 给文章分 lane
  • 做启发式评分
  • 选出最值得代表文风的样本
  • 再把这些东西压成一份更轻的 runtime

这套东西继续塞在 skill 里，维护成本会越来越高。它更像一个需要被重复运行、可测试、可恢复、可替换 provider 的构建系统。

所以最后落成 Python 项目，其实是很自然的结果。不是为了“更工程化”而工程化，而是因为它真的已经长成了一条 pipeline。

它真正省 token 的地方，不是一个点，而是三层拆分

现在回头看，blog-style-suite 省 token 的关键，不是某一个优化技巧，而是架构上主动把问题拆开了。

第一层，扫描和筛选尽量不用 AI

在 scanner.py 这一层，很多事情本来就没必要上模型。

比如：

  • frontmatter 解析
  • <!--more--> 检查
  • 正文是否过短
  • 是否嵌了 prompt
  • blockquote 占比是否过高
  • 路径遍历、标题提取、小标题提取

这些东西交给规则做就行，没必要每篇文章都调一次模型。

后来在 bc4b950 这个提交里，scan 阶段干脆彻底改成了纯启发式。提交说明里写得很直接，AI 调用从 ~2000 次降到 5 次/provider。这不是小修小补，是整条成本曲线都被改了。

第二层，把“每篇一调”改成“每类一调”

真正贵的地方，不只是有没有 AI，而是调用粒度。

如果每篇文章都单独做 AI 筛选、AI 分类、AI 打标签，那文章一多，调用次数就会直接爆掉。后面重构以后，思路变成：

  • scan 阶段先本地启发式处理
  • build 阶段再按 lane 批量精选
  • 每个 lane 一次
  • 最后作者风格压缩再来一次

这就把原来按文章线性增长的成本，压到了按 lane 增长。lane 就四个，作者风格再加一次，冷启动也就 5 次左右。

这一步其实很关键。很多人说“优化 token”，最后只是从大 prompt 改成小 prompt。但更本质的优化，是把调用单位改了。

第三层，运行时只读压缩后的 runtime

这也是我觉得最重要的一层。

现在 blog-writer 正常写稿时，不再去读一堆原始语料，而是只读 .agents/data/blog-writing/published.runtime.json。这个文件顶层结构已经被压到比较稳定的几块：

  • author_style
  • lanes
  • samples
  • writer_guide

这意味着写稿阶段拿到的，不再是一堆原始文章，而是一份预处理后的、已经筛好、压好的“文风索引”。

这才是预处理真正省 token 的地方。不是说历史文章没用了，而是它们的作用，已经提前在构建阶段被消化了一轮。

数据结构为什么要这么设计

我觉得这套结构设计得比较对路的地方，是它没有试图做成一个巨型万能 schema，而是刻意分成了几种不同粒度的数据。

author_style 负责全局声音。比如开头习惯、结构习惯、推理习惯、事实处理方式。这个层是“这人整体怎么写”。

lanes 负责局部增强。技术、金融、随笔、工具体验，这几类题材虽然都是同一个作者写，但确实会有局部差异。这个层是“这类题更适合怎么写”。

samples 不是全文，而是 sample card。里面只有开头卡、结构卡、声音卡、不要抄什么。这个层是“拿几个最能代表风格的例子做提示”。

writer_guide 再去约束读取顺序和读取上限。这个层是“运行时别贪多，读到够用就停”。

你会发现，这几个层级其实刚好对应四个问题：

  • 我整体是什么风格
  • 这篇当前属于哪类
  • 有没有几个高价值样本可以参考
  • 运行时怎么读才不浪费

这比把所有历史文章硬塞给模型清晰得多。

checkpoint 和缓存，不只是省时间，也是在省 token

后面又做了两件我觉得非常务实的事。

一个是 provider checkpoint 恢复。2e6bb9f 这个提交里，把 provider 的中间结果恢复机制补上了。这样构建中断以后，不需要全部重来。

另一个是 screening / lane 指纹缓存复用。5b82d23 这个提交里，用指纹判断旧结果还能不能复用。条件没变，就直接拿旧结果，不再重跑。

这两步表面上是“工程体验更好”，本质上还是在控成本。因为对接本地模型也好，对接在线模型也好，只要你能少调一次，就是真实省下来一次。

所以核心设计思路其实很简单

如果要用一句话总结 blog-style-suite 的设计思路，我会这么说：

把高频阶段做轻，把低频阶段做重，把需要模型判断的地方压缩到最少，把不需要模型判断的地方尽量本地化。

这套东西说白了不神秘。真正难的不是“想到可以预处理”，而是愿不愿意把整条链路拆清楚。

很多时候，AI 工作流一开始很丝滑，是因为你还没到必须算账的时候。等历史数据多起来、文章数量多起来、provider 也多起来，迟早会碰到成本问题。

这时候如果前面没有做数据分层，没有把扫描、构建、运行时拆开，那后面几乎必然会越来越重。

所以下一篇，我就继续把这个问题往前推一步。

既然风格构建也贵，那为什么不接本地模型？本地模型接进来以后，和在线版本到底差在哪？Minimax 放在这条链路里，又扮演了什么角色？

原始提示词 

本次的内容比较多，拆分成系列文章：去年就有很多稿子是通过大模型写的，那会是自己写个大纲或者问题清单，然后AI出稿子，复制内容到本地 md 文档，填写头信息，标签信息、发布文章；近期 codex 用了很多，发现 codex 里面的联网搜索能力很强，那我是不是能写个 skill，将这些事情自动化，此时诞生了 skill blog-writer 的第一稿，我还想着让 AI 学习我以前文章的风格，这就导致 blog-writer 运行的时候，很费 token，后续我针对 blog-writer 进行了多个版本的优化，拆分了 数据模块，数据生成的模块，原本数据生成的模块还是独立的 skill，写着写着，我就发现，更适合做成 Python 项目，此时就有了 blog-style-suite，然后我又发现，训练风格数据，也是比较费 token，我就想着用本地的大模型，对接了本地的大模型，我又想到了对比下本地大模型和在线版本的区别，又对接了 minimax；blog-style-suite 和 blog-writer 的演化历史可以分析的 git 提价记录。顺带基于本地 blog-writer、blog-style-suite 的代码，可以讲讲里面的设计思路，是如何做到了节约 token，数据结构是如何设计的，核心的设计思路。Token 富裕完全能生吃历史文章，预处理能节约很多 token

写作思路摘要

  • 这一篇重点放在架构拆分和 token 成本，不重复第一篇的流程起源。
  • 直接用三层拆分来解释节省 token 的原因，比泛泛说“做了优化”更清楚。
  • 把 scanner、builder、runtime 的职责分开写，顺手把数据结构设计讲透。
  • 特意把 checkpoint 和指纹缓存单独拎出来，因为它们体现的是“工程化以后才有的现实收益”。
