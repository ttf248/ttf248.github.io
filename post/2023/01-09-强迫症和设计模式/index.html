强迫症和设计模式

2023-01-09
计算机

文章目录

    • 举个栗子

设计的基本几个原则，常见的设计模式，都了解过，作为实战派，缺乏足够的迭代过程 些许的强迫症，导致了项目初期，特别是脱离框架的时候，容易搞事情，拖累进度

C++ 做项目开发，金融IT的项目，也没个框架，从零就开始起步，有点草台班子的意思

恒生其实有个T2的服务端插件框架，用起来很别扭，基础的轮子，对外暴露的接口，很多都还停留在十年前 组里也不限制发挥，Github 搞起，见识了很多优秀的轮子，拿来就用，拼凑凑就开工了 设计类似的业务还行，折腾了一圈，总归有些收获，设计新业务的时候，就容易跑飞

毕业就想着有个大佬手把手带我，没想到，还是自己一个人，自己摸索，师傅也就一年，没多久就离职了

又开始碎碎念了，回到主题，设计模式，裸着看概念都很好理解，脱离项目的那种

翻阅过，代码整洁之道，还没持续多久，就被公司的事情打断了，那么多年，完整看过的书，几乎没有

有点遗憾，但是，也没办法，工作就是这样，不是吗？

我本一介凡人，奈何有个强迫症，总是想着，能不能把这些概念，用到项目中，用到实际的业务中

举个栗子

服务启动需要一份配置文件，通过配置，能控制服务的日志级别、监听的端口、处理哪些市场的数据，tcp缓存的buff、内部队列的最大长度 配置文件是个单独的对象，设计成单例，通过GetInstance()获取，配置文件的内容，通过GetConfig()获取，内置命名空间，函数的静态成员

quickfix + spdlog，能设计出来一份优雅的代码

没错，就是优雅，如果从未阅读 Github 那些轮子的 Readme，我不会觉得自己的代码有问题，见过优雅的设计，就不愿再低头

网卡收到了数据，扔到业务层面处理，数据包含不同市场、单个线程处理全部市场？如果需要灵活控制，每个市场独立的线程呢？ 不同市场数据的映射、数据的格式化、时间戳的处理，繁琐的业务逻辑放进来，逻辑开始撕裂
